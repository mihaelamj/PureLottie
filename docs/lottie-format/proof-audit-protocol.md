# Proof Audit Protocol and Red-Team Checklist

Status: `theorem` (The audit protocol defines the formal verification and classification rules for conformance claims)

This document defines the written adversarial audit checklist and red-team protocol used to certify conformance, correctness, and reversibility claims in PureLottie. To prevent confidence-only closure, no claim may be marked as `witnessed` or `theorem` unless it satisfies every item in the checklist.

---

## 1. The Audit Checklist

For every correctness, conformance, or reversibility claim, the audit checklist requires:

1. **Equivalence Relation Stated**: The exact mathematical or semantic relation between the compiler's output and the reference must be formally defined (e.g., $|v_{\text{actual}} - v_{\text{expected}}| \le \epsilon$ for a derived tolerance $\epsilon$, or exact structural identity).
2. **Witness Evidence Present**: A committed evidence file (such as a lottie-web reference trace under `lottie-web-intent/`, a reversibility report, or a confluence witness) must exist in the repository to back the claim.
3. **Tolerance Derived**: If the claim is numeric, the tolerance threshold must carry a mathematical derivation from a stated arithmetic model in [Tolerance Derivations](tolerance-derivations.md). If it is not derived, it must be explicitly labeled as `assumed`.
4. **Fault Injection Calibrated**: A controlled fault (mutation) in the catalog must be active, and a calibration test in [LottieFaultInjectionCalibrationTests](file:///Volumes/Code/DeveloperExt/public/PureLottie/Tests/LottieEvaluationTests/LottieFaultInjectionCalibrationTests.swift) must verify that the gate fails under the mutation on a fixture exercising it.
5. **Evidence Pinned and Reproducible**: The evidence must be generated by a pinned, documented toolchain process (Playwright, browser revision, content hash, and update command). If the process is not fully pinned, the evidence cannot claim full reproducibility.
6. **Counterexample Bank Checked**: Negative or counterexample tests must verify that values outside the accepted interval, or malformed inputs, are actively rejected by the validator or difference gate.

---

## 2. Downgrading Policy

A claim that fails to satisfy any of the checklist criteria must be downgraded:

* **From `witnessed` or `theorem` to `sampled`**: If the claim has evidence but lacks a formal confluence proof, totality proof, or derived tolerance.
* **From `witnessed` to `assumed`**: If the tolerance is empirically fudged instead of mathematically derived, or if the reference-engine APIs do not supply a portable error bound.
* **From `witnessed` to `blocked`**: If the generation process is unreproducible, if a dependency is unpinned, or if a known coverage hole exists.

No claim may be closed or tagged as complete in the documentation or code while carrying a status of `assumed` or `blocked` without naming the specific missing piece and creating a tracking issue.

## 4. Red-Team checklist for PR Review

Every incoming pull request must be audited by the red-team checklist:

1. Did the change add any unclassified observed Lottie field to the parser? If yes, it violates rule **C6** (Model-or-Report totality).
2. Does the tolerance change have an arithmetic derivation? If it was changed empirically, it must be downgraded to `assumed`.
3. Was a new compiler edge added without a termination argument? If yes, it violates rule **K2** (termination cost analysis).
4. Does the verifier target share any evaluation logic with the compiler? If yes, it violates rule **K5** (verifier size and independence).
5. Were any tests skipped or disabled? If yes, it violates rule **0.3** (zero skipped tests).
